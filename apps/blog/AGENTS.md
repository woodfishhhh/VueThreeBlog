# AGENTS.md — apps/blog

Blog SPA migrated from Vue 3 + vite-plus to **Nuxt 5 nightly** (Vite 8 + Rolldown).

## Commands

Run from this directory (`apps/blog`) or via root workspace scripts.

```bash
# Development
nuxt dev                       # dev server on http://localhost:3000
nuxt prepare                   # regenerate .nuxt/ type declarations

# Content pipeline (must run before build if content changed)
npm run generate:content       # tsx content-tools CLI: parse Markdown + optimise images
npm run generate:content:ci    # CI variant: reuse existing optimised assets
npm run optimize:images        # optimise images only (skip Markdown parse)

# Build
nuxt generate                  # static SPA → dist/
npm run build:deploy           # prod build: base=/newBlog/, then verify-dist

# Quality
nuxt typecheck                 # vue-tsc type check
vitest run                     # unit tests
vitest watch                   # watch mode
playwright test                # E2E tests (requires built/running server)
```

## Nuxt 5 Configuration (`nuxt.config.ts`)

Key settings:

| Setting | Value | Reason |
|---|---|---|
| `srcDir` | `"src"` | Keeps existing directory layout |
| `ssr` | `false` | SPA mode — Three.js requires browser APIs |
| `compatibilityDate` | `"2025-01-01"` | Required by Nuxt 5 |
| `modules` | `["@vite-pwa/nuxt"]` | PWA support |
| `css` | `["~/assets/main.css"]` | Global CSS entry |
| `postcss.plugins` | `{ "@tailwindcss/postcss": {} }` | Tailwind v4 via PostCSS (not `@tailwindcss/vite`) |
| `nitro.output.publicDir` | `"dist"` | Matches old Vite output path |

**Why PostCSS instead of `@tailwindcss/vite`?**  
Importing `@tailwindcss/vite` at the top of `nuxt.config.ts` causes a CJS/ESM conflict when Nuxt's jiti evaluates the config. The PostCSS plugin avoids this.

**Why no `@pinia/nuxt` module?**  
`@pinia/nuxt@0.11` requires `nuxt ^3||^4`. Pinia 3 is registered manually via `src/plugins/pinia.ts`. Auto-imports (`defineStore`, `storeToRefs`, `acceptHMRUpdate`) are declared in `imports.presets`.

## Routing

Multi-panel home: `/`, `/blog`, `/works`, `/author`, `/friend` all render `HomeView.vue`. The active panel is driven by `SitePanelMode` Pinia store, updated by `plugins/site-mode.client.ts` on every navigation.

`/posts/:slug` route uses the `resolve-post-slug` middleware to redirect alias slugs to canonical ones.

Route names are defined via `definePageMeta({ name: "xxx" })` in each page file to preserve backward-compatible `<RouterLink>` usage.

## Content Pipeline

Markdown posts live in `content/posts/`. Run `generate:content` to:
1. Parse Markdown + frontmatter → `src/generated/post-index.json`
2. Copy/optimise images → `public/content-assets/`
3. Output individual post JSON files to `public/content/`

Generated files must NOT be committed except via the CI content-generation step.

## PWA

`@vite-pwa/nuxt` with Workbox `generateSW` strategy. The service worker precaches all dist assets. Config lives in `nuxt.config.ts` under `pwa:`.

## Testing

- Unit tests: `tests/` directory, `vitest.config.ts`, jsdom environment
- E2E tests: `tests/e2e/` with Playwright (run against `nuxt preview` or deployed URL)
- Setup file: `tests/setup.ts`

## Important Notes

- **Never manually edit** `src/generated/` — content-tools owns those files.
- **Tailwind v4** has no `tailwind.config.js`. All custom tokens go in `@theme {}` blocks in `main.css`.
- **Three.js** components must remain client-only; `ssr: false` handles this globally.
- **The `@/` alias** maps to `src/` (same as `~/`). Both work.
